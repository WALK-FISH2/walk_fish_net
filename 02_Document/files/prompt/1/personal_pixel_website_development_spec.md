# 个人像素叙事网站开发需求文档

> 文档用途：直接交给 Codex 作为项目开发需求、技术设计和验收依据。  
> 项目定位：纯前端、可静态部署的个人网站，用于展示博文、个人项目和可交互程序。  
> 暂定站点栏目：**首页 / 文章 / 做点啥呢 / 关于我**。  
> 核心视觉：高精细二维像素风、复古游戏氛围、滚动驱动的连续场景叙事。

---

## 1. 项目背景

我要制作一个个人网站，主要用于：

1. 发布和展示个人博文；
2. 展示我开发过的程序、项目和前端演示；
3. 形成具有个人辨识度的作品集；
4. 通过滚动交互，让访问者像游玩一段复古游戏关卡一样浏览网站。

网站第一阶段必须是纯前端项目，不依赖自建后端、数据库或长期运行的服务器进程。构建后应生成可直接部署到 GitHub Pages、Cloudflare、Netlify、Vercel 等静态托管平台的文件。

需要注意：“纯前端不需要自建服务器”不代表网页不需要托管。最终仍然需要静态托管平台或对象存储/CDN 对外提供 HTML、CSS、JavaScript、图片和字体文件。

---

## 2. 项目目标

### 2.1 核心目标

- 建立一个可长期维护的个人内容网站；
- 博文和项目内容可以通过 Markdown/MDX 文件添加，不需要修改页面组件；
- 首页具有类似创意工作室官网的沉浸式滚动体验；
- 页面滚动过程不是简单的淡入淡出，而是一个连续、可逆、自然衔接的场景变化；
- 视觉上体现精致的二维像素美术，而不是粗糙方块、体素或“我的世界”风格；
- 桌面端体验完整，移动端保留核心视觉但适当降低粒子和渲染复杂度；
- 所有页面均可以静态构建和部署。

### 2.2 非目标

第一阶段不实现以下功能：

- 用户注册、登录；
- 后台内容管理系统；
- 数据库存储；
- 服务端评论系统；
- 浏览量统计接口；
- 需要服务端运行的程序；
- 在线支付；
- 实时多人交互。

如以后需要评论、统计、表单或搜索，可以接入第三方服务或 Serverless Functions，但第一阶段不能因此引入传统后端。

## 3. 风格定义

### 3.1 “32bit/64bit 像素风”的实际定义

这里的“32bit/64bit”不是网页图片的颜色位深，也不是严格的游戏美术分类，而是用于描述以下视觉目标：

- 比红白机早期 8-bit 像素画更精细；
- 具有 16-bit/32-bit 主机时代的丰富色彩、层次、光影和动画；
- 可以出现较大的 32×32、48×48、64×64、96×96 像素精灵；
- 场景中允许存在渐变、半透明粒子、光晕和水面效果；
- 保留清晰的像素边缘，不使用模糊的插值放大；
- 画面总体为二维横版游戏美术，不使用三维方块堆叠和体素建模。

### 3.2 风格参考原则

可以参考经典横版平台游戏的明快配色、关卡节奏、云朵、山丘、砖块、金币、旗帜、像素 UI 等表现方式，但必须满足：

- 不直接使用超级玛丽、任天堂或其他游戏的角色、图标、地图、音效、Logo 和原始素材；
- 不一比一复刻现有关卡；
- 不使用《加勒比海盗》的影片画面、音乐、船只设计或受版权保护的素材；
- 仅借鉴“复古平台游戏”和“海洋平滑过渡进入另一个世界”的抽象氛围；
- 所有角色、图标、场景和音效应原创，或使用授权清晰的开源素材并保留授权说明。

### 3.3 视觉关键词

- 复古；
- 可爱；
- 精致像素；
- 冒险；
- 明亮陆地；
- 深海；
- 浪花；
- 平滑过渡；
- 星空；
- 神秘；
- 连续世界；
- 轻游戏化；
- 不幼稚；
- 不粗糙；
- 不体素化。

## 4. 信息架构

### 4.1 页面路由

建议实现以下路由：

```text
/                     首页：沉浸式滚动叙事
/articles             文章列表
/articles/[slug]      文章详情
/projects             “做点啥呢”项目列表
/projects/[slug]      项目详情
/about                关于我
/404                  404 页面
```

页面导航中显示：

```text
首页
文章
做点啥呢
关于我
```

“做点啥呢”是对外展示名称，代码、目录和数据类型中统一使用 `projects`，不要在代码中使用中文拼音作为核心模块名。

### 4.2 首页和内容页的关系

首页负责“体验和引导”，文章页与项目页负责“阅读和查看详情”。

首页只展示：

- 精选文章 3～4 篇；
- 精选项目 3～6 个；
- 简短的个人介绍；
- 进入文章列表和项目列表的入口。

首页不能把完整文章正文塞入 Canvas。文章标题、摘要、按钮和导航必须使用真实 DOM 元素，保证文字可复制、可聚焦、可被搜索引擎识别。

## 5. 首页整体叙事结构

首页是一条连续的纵向滚动旅程，由三个主场景和两个过渡段构成。

### 5.1 推荐滚动长度

桌面端首页总滚动长度建议为 `800vh～1000vh`。

建议初始值：

```text
总长度：900vh
第一阶段：0%～30%
陆地到深海过渡：30%～38%
第二阶段：38%～66%
深海到星空过渡：66%～80%
第三阶段：80%～100%
```

所有动画必须由统一的标准化进度 `progress: 0～1` 驱动。

不得只监听“进入某个区块”然后播放一次不可逆动画。用户向上滚动时，场景必须按照相同进度自然倒放。

### 5.2 页面固定结构

首页采用以下结构：

```text
- 固定全屏视觉画布 Canvas
- 固定或半固定的顶部导航
- 正常文档流中的滚动触发区
- 叠加在 Canvas 上的 DOM 内容卡片
- 右侧或底部的像素进度指示器
- 降级模式和减少动画模式
```

Canvas 负责：

- 天空、云、山丘、海水、鱼群、浪花、星空；
- 粒子；
- 前景与背景的视差；
- 像素精灵；
- 场景颜色、光束、透明度和粒子形态过渡。

DOM 负责：

- 站点标题；
- 文章标题、摘要和链接；
- 项目名称、简介和按钮；
- 导航；
- 辅助说明；
- 无障碍文本。

## 6. 第一阶段：明亮像素陆地

### 6.1 场景定位

第一阶段是访问者进入网站后看到的主世界。整体明亮、轻松、色彩鲜艳，类似复古横版平台游戏的冒险开场，但不能直接复刻任何现有游戏。

### 6.2 场景元素

背景分层建议：

1. 天空渐变；
2. 远处云层；
3. 远山和圆润丘陵；
4. 中景树木、房屋或塔；
5. 近景草地、岩石、花朵；
6. 前景像素草叶、尘土和小粒子。

可加入一个原创的“小旅行者”角色作为视觉引导，但角色不是必须项。若加入角色：

- 角色大小建议为 48×48 或 64×64 逻辑像素；
- 具有待机、行走、抬头、下潜等动作；
- 不能设计成现有游戏角色的换色版本；
- 不使用用户本人的肖像，除非后续单独提供素材。

### 6.3 首屏内容

首屏必须包含：

- 用户姓名或个人品牌名称占位符；
- 一句个人定位；
- 两个入口按钮：“看看文章”“看看我做了什么”；
- 向下滚动提示；
- 像素风加载完成后的简短入场动画。

默认占位文案：

```text
你好，我是 [姓名]
软件开发者，也喜欢把一些奇怪的想法做成东西。

[看看文章]  [做点啥呢]
```

### 6.4 滚动行为

在第一阶段中：

- 云朵以最慢速度移动；
- 远山速度略快；
- 地面和近景速度最快；
- 角色可以跟随滚动缓慢向右移动；
- 页面标题逐渐缩小并移向左上角；
- 精选文章以路牌、公告板、关卡入口或漂浮窗口的形式依次出现；
- 卡片进入时采用 4～8 像素的分块显现、像素擦除或抖动显现；
- 禁止只使用普通透明度淡入作为全部交互。

### 6.5 第一阶段色板

建议使用以下基础色板，可在实现中建立为 CSS 变量：

```css
--land-sky: #73D7FF;
--land-sky-light: #C8F2FF;
--land-cloud: #FFF6DD;
--land-grass: #58C65A;
--land-grass-dark: #267A40;
--land-soil: #A85F3F;
--land-accent-red: #F45B5B;
--land-yellow: #FFD25A;
--land-ink: #18233B;
```

## 7. 第一阶段到第二阶段：下潜过渡

### 7.1 过渡目标

陆地进入深海不能直接切换背景，也不能突然全屏变蓝。需要构造“世界逐渐被水淹没，然后镜头向海底下潜”的连续过程。

### 7.2 过渡流程

建议按以下顺序实现：

1. 天空亮度略微下降；
2. 风变强，云层加速；
3. 画面底部出现水面反光；
4. 水面逐渐上涨；
5. 水线经过 DOM 卡片时，卡片短暂发生折射或像素错位；
6. 水面覆盖整个屏幕后，出现气泡；
7. 镜头继续向下，水面逐渐远离画面顶部；
8. 陆地元素变成远处剪影并消失；
9. 色板从青蓝逐渐过渡到深蓝。

水面经过页面时，要有清晰但不影响阅读的折射效果。不能持续让文字扭曲。

---

## 8. 第二阶段：深邃海底

### 8.1 场景定位

第二阶段的情绪从轻松明亮转向安静、深邃、神秘。画面需要明显变暗，但不能黑到看不清信息。

### 8.2 场景元素

背景分层建议：

1. 远处深海渐变；
2. 水面方向透下来的微弱光束；
3. 缓慢移动的鱼群剪影；
4. 近处海草和珊瑚；
5. 气泡；
6. 少量发光水母或生物荧光粒子；
7. 沉入海底的抽象遗迹、终端或数据碎片。

不要使用写实恐怖海怪。整体应保持诗意和可爱，而不是恐怖游戏。

### 8.3 内容展示

“做点啥呢”项目预览主要放在第二阶段。

项目卡片可以设计为：

- 发光气泡；
- 舷窗；
- 海底终端；
- 漂浮的像素显示器；
- 贝壳或遗迹中的数据面板。

每个项目卡片至少显示：

- 项目名称；
- 一句话简介；
- 技术栈标签；
- 项目状态；
- 详情按钮；
- 在线演示按钮；
- 源码按钮，若没有链接则不显示。

卡片不能完全画在 Canvas 中。文字和按钮仍需用 DOM 实现，Canvas 只提供容器背景、粒子和装饰。

### 8.4 交互细节

- 鼠标移动时，附近气泡产生轻微偏移；
- 项目卡片悬停时边框产生像素流光；
- 点击项目卡片进入项目详情页；
- 项目卡片应支持键盘 Tab 聚焦；
- 触摸端取消依赖 hover 的必要信息；
- 气泡运动采用轻微循环，不要产生明显晕动感。

### 8.5 第二阶段色板

```css
--sea-top: #0B5275;
--sea-mid: #092E55;
--sea-deep: #04162F;
--sea-cyan: #25BEC4;
--sea-light: #89F0D7;
--sea-coral: #FF806E;
--sea-text: #E5FAFF;
--sea-muted: #8DB9CB;
```

## 9. 第二阶段到第三阶段：浪花翻涌与海空平滑过渡

这是整个网站最重要的视觉段落。

### 9.1 核心要求

需要表现“海洋翻涌，并通过颜色、光束、透明度和粒子形态连续进入星空”的感觉。背景世界在整个过程中保持水平，这里不出现船，也不复刻电影镜头。

过渡必须让用户感觉：

1. 海水开始发生异常；
2. 浪花和水面逐渐占据视觉中心；
3. 世界上下方向保持稳定，深海色和水下光束连续变化；
4. 海水中的泡沫、气泡和高光自然转化为星星；
5. 最终进入一个宁静的星空世界。

### 9.2 建议动画流程

#### A. 平静被打破

- 深海中的气泡速度提高；
- 海草朝同一方向弯曲；
- 远处光束发生倾斜；
- 水流粒子逐渐形成旋涡；
- 水流与光束方向轻微变化，但背景世界保持水平。

#### B. 向水面上升

- 镜头快速但平滑地向上；
- 背景深蓝逐渐变亮；
- 水面反光由顶部接近中心；
- 鱼群和海底元素向下远离；
- 项目卡片逐渐退出，但固定导航保持稳定。

#### C. 浪花翻涌

- 水面在画面中心形成明显的像素浪峰；
- 使用多层波形、泡沫粒子、白色像素块和透明水雾；
- 波浪不能只是一个正弦曲线；
- 至少包含前浪、中浪、远浪三层；
- 浪花粒子数量根据设备性能动态调整；
- 波浪遮挡场景接缝，避免颜色直接跳变。

#### D. 固定世界方向与平滑转场

- Canvas 背景世界在整个海洋到星空过程中固定为 0°；
- 导航、关键正文 DOM 和背景世界始终保持水平；
- 不得使用任何角度的世界旋转、翻转、镜面翻转或整体倾斜；
- 通过颜色、光束、透明度、浪花和粒子形态继承完成连续过渡；
- 桌面、移动端和“减少动态效果”模式均遵守相同的固定方向约束。

#### E. 海水转化为星空

这是衔接自然的关键，应实现元素之间的形态继承：

- 白色泡沫像素逐渐散开并成为星点；
- 气泡逐渐缩小、亮度提高，转化为星尘；
- 水面高光拉长后成为银河；
- 深海背景色保持为深蓝，再慢慢过渡到靛紫和黑色；
- 波浪边缘消散后，露出星云；
- 水流旋涡继续保留运动方向，逐渐成为星云旋涡；
- 最终完全看不到海水，但用户可以理解星空来自刚才的海洋。

### 9.3 防晕动要求

- 页面 UI、Canvas 背景世界和固定导航始终保持水平；
- 不得使用整体旋转、翻转或倾斜制造转场；
- 不使用持续镜头抖动；
- 不强制平滑滚动或锁死滚轮；
- 用户停止滚动时，动画应停在对应进度，不自行高速播放；
- 支持 `prefers-reduced-motion`。

## 10. 第三阶段：星空世界

### 10.1 场景定位

第三阶段是旅程结束后的开阔区域。整体宁静、深邃，并带有“进入另一个世界”的完成感。

### 10.2 场景元素

- 多层星点；
- 远处星云；
- 缓慢旋转的像素星团；
- 流星；
- 星座连线；
- 小行星、卫星或原创漂浮岛；
- 从上一阶段转化而来的泡沫星尘。

### 10.3 内容安排

第三阶段主要展示：

- 关于我简介；
- 技能和关注方向；
- 联系方式；
- GitHub、邮箱和其他社交链接；
- “重新开始旅程”按钮；
- 完整导航入口；
- 页脚。

精选文章和项目可以在星空中形成星座：

- 每颗主星对应一篇文章或一个项目；
- 悬停时显示名称；
- 点击后进入详情页；
- 移动端改为普通列表，不强制使用星座点击。

## 11. 导航与通用交互

### 11.1 顶部导航

桌面端：

- 左侧为个人 Logo 或姓名；
- 右侧为首页、文章、做点啥呢、关于我；
- 导航背景随场景变化；
- 陆地阶段为浅色像素面板；
- 深海阶段变为半透明深蓝；
- 星空阶段变为深色星尘边框；
- 导航切换颜色时保持足够对比度。

移动端：

- 使用像素风菜单按钮；
- 打开后显示全屏或抽屉式菜单；
- 菜单必须能通过 ESC 关闭；
- 打开菜单时锁定背景滚动；
- 焦点限制在菜单内。

### 11.2 滚动进度

实现一个像素关卡式进度条，显示三个节点：

```text
陆地 → 深海 → 星空
```

要求：

- 当前阶段高亮；
- 点击节点可平滑跳转到对应阶段；
- 进度条与背景世界始终保持水平；
- 支持键盘操作；
- 在移动端缩小为底部三个像素点或短条。

## 15. 技术方案

### 15.1 推荐技术栈

建议使用：

```text
Astro
TypeScript
React（仅用于首页交互岛）
GSAP + ScrollTrigger
PixiJS
CSS Modules 或原生 CSS
Markdown / MDX
```

选择原则：

- Astro 负责静态页面、文章、项目详情和构建输出；
- React 只负责需要复杂状态或生命周期的首页交互场景；
- GSAP ScrollTrigger 负责把滚动进度映射到时间轴；
- PixiJS 负责二维像素精灵、粒子、海水和星空；
- 普通文本和卡片保持 DOM 渲染；
- 不使用完整 React SPA 承担所有内容页，避免不必要的 JavaScript 和路由问题；
- 不使用 Three.js 构建体素世界；
- 不依赖后端。

### 15.2 渲染架构

建议实现一个统一的场景控制器：

```ts
interface SceneController {
  init(container: HTMLElement): Promise<void>;
  resize(width: number, height: number, dpr: number): void;
  update(progress: number, deltaTime: number): void;
  setQuality(level: 'low' | 'medium' | 'high'): void;
  destroy(): void;
}
```

场景模块：

```text
OverworldScene
DiveTransition
UnderwaterScene
OceanToSpaceTransition
SpaceScene
```

统一入口接收标准化进度：

```ts
updateStoryProgress(progress: number)
```

场景内部自行把全局进度映射到局部进度：

```ts
const local = clamp((progress - start) / (end - start), 0, 1)
```

### 15.3 动画原则

- 滚动动画采用单一主时间轴；
- 所有关键动画可逆；
- 不在多个组件中各自读取滚动位置造成冲突；
- 环境循环动画与滚动进度分离；
- 随机粒子使用固定种子，避免滚动回退时画面跳变；
- 避免大量 DOM 元素逐帧更新；
- 仅对 `transform`、`opacity` 和 Canvas 内容做高频动画；
- 不使用 scroll 事件直接进行重计算，使用 GSAP ticker、requestAnimationFrame 或受控更新；
- 页面不可劫持滚轮；
- 禁止为了“顺滑”而让滚动明显滞后于用户输入。

### 15.4 像素渲染要求

```css
.pixel-art {
  image-rendering: pixelated;
  image-rendering: crisp-edges;
}
```

Canvas 要求：

- 关闭纹理平滑；
- 设置精灵纹理为 nearest-neighbor；
- 内部逻辑分辨率低于物理屏幕分辨率，再整数倍放大；
- DPR 最大值限制为 2；
- 尽量保证精灵坐标落在整数像素；
- 禁止把低分辨率素材用双线性插值放大；
- 大面积渐变和光晕可以平滑，但精灵边缘必须清晰。

建议桌面逻辑分辨率：

```text
640×360 或 960×540
```

根据屏幕比例裁切或扩展背景，不能拉伸角色和精灵。

目标宽度：

```text

```

调整：

- 减少前景层；
- 减少粒子；
- 卡片改为单列或双列；

```text
< 768px
```

- 取消高成本折射 Shader；
- 项目卡片使用纵向列表；
- 星座改为可读列表；
- 世界方向保持 0°，通过简化颜色、光束和粒子形态过渡；
- 保证按钮至少约 44px 可触控尺寸；
- 不出现横向滚动条；
- 文字不被安全区域或浏览器底栏遮挡。

## 18. 性能要求

### 18.1 加载策略

- 首屏只加载第一阶段必要素材；
- 深海和星空素材延迟加载；
- 使用纹理图集；
- 图片优先使用 WebP 或 AVIF，并保留必要 PNG；
- 非首屏图片懒加载；
- 字体子集化或只保留需要字重；
- 不使用全屏背景视频作为主要实现；
- 资源加载失败时仍能显示内容和导航。

### 18.2 性能目标

目标而非绝对硬限制：

- 普通桌面设备尽量维持 60fps；
- 中低端手机不低于可接受的 30fps；
- 首屏压缩资源尽量控制在 1.5MB 左右；
- JavaScript 按场景拆包；
- 页面切后台后暂停非必要动画；
- Canvas 不可见时降低更新频率；
- 监听 WebGL context lost 并提供恢复或降级；
- 不产生持续增长的内存占用；
- 页面切换时正确销毁 PixiJS 场景、Ticker、监听器和 ScrollTrigger。

### 18.3 自动画质

根据以下条件选择画质：

- 设备像素比；
- 屏幕宽度；
- `navigator.hardwareConcurrency`；
- 是否为触摸设备；
- 初始化帧率采样；
- 用户是否开启减少动画。

画质等级：

```text
high：完整粒子、折射、三层浪花
medium：减少粒子、简化折射
low：CSS 渐变背景、少量 Canvas 粒子、无复杂 Shader
```

## 31. 最终效果概述

访问者进入网站时，首先看到一个明亮、可爱、精细的像素世界。随着向下滚动，天空、云朵、丘陵和文章入口产生多层视差，像在游玩一段复古横版关卡。

继续向下，水面从画面底部上涨，世界逐渐被海水覆盖。镜头进入海底，颜色变得深邃，气泡、鱼群和发光生物围绕项目卡片缓慢移动，“做点啥呢”模块像一组沉在海底的个人作品档案。

再继续滚动，海流变强，镜头向水面上升。浪花在画面中央翻涌，背景世界始终保持水平。白色泡沫散开成为星点，气泡转化为星尘，水面高光延伸为银河，海洋的深蓝自然过渡为星空的靛紫。

最终，访问者进入安静辽阔的星空，看到关于我、联系方式和由文章与项目组成的星座。整个体验可以向上滚动倒放，也可以通过导航直接进入普通文章和项目页面。

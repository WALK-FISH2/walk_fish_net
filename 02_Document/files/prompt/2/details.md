## 第一阶段：明亮陆地世界

将当前首屏的纯蓝背景替换成完整的明亮像素陆地。

必须包含：

1. 天空渐变；
2. 至少两层云；
3. 远处山丘；
4. 中景树木、建筑或塔；
5. 近处草地；
6. 草叶、花朵或尘土粒子；
7. 一个原创的小型像素旅行者，或其他原创的视觉引导物；
8. 前景遮挡元素。

建议画面结构：

```
顶部：天空、云层
中部：远山、丘陵、塔
底部：草地、道路、小花
前景：草叶或岩石
```

滚动效果：

- 云层移动最慢；
- 远山稍快；
- 地面和前景移动更明显；
- 小旅行者随着滚动缓慢向右移动；
- 文章内容依次出现在道路沿线；
- 向上滚动时所有动作必须自然倒放。

文章卡片不要继续使用普通大矩形。

把三篇精选文章设计成：

- 像素路牌；
- 公告板；
- 关卡入口；
- 竖立在草地上的信息牌。

文章标题和摘要仍然使用DOM，但外框和装饰应融入陆地场景。

首屏个人介绍面板可以保留，但需要调整：

- 面板尺寸适当缩小；
- 不要遮挡大部分场景；
- 增加像素窗口角标；
- 去掉过度现代化的巨大排版感；
- 与陆地环境形成层次，而不是悬浮在纯色背景上。

## 四、陆地到深海的连续下潜

不能在滚动到某个点时直接把背景颜色切换为深蓝。

实现以下连续过程：

1. 天空逐渐变暗；
2. 云层速度略微提高；
3. 页面底部出现水面；
4. 水面带有反光和轻微波纹；
5. 水位随着滚动不断上升；
6. 水线经过文章卡片时产生短暂像素错位或折射；
7. 水覆盖全屏；
8. 出现气泡；
9. 镜头继续向下；
10. 水面逐渐远离到屏幕顶部；
11. 陆地元素变成模糊剪影并消失；
12. 色调自然过渡为深海蓝。

水位上涨必须由滚动progress控制，并且可以向上滚动倒放。

不要使用一次性播放的视频。

------

## 五、第二阶段：深海项目世界

当前深海阶段仍然是亮蓝色背景，需要彻底重做。

深海必须包含：

1. 从上到下逐渐加深的海水；
2. 顶部透下来的光束；
3. 至少两层鱼群剪影；
4. 气泡；
5. 海草；
6. 珊瑚或海底岩石；
7. 少量发光水母；
8. 生物荧光粒子；
9. 远处遗迹或数据终端；
10. 底部海床；
11. 前景海草遮挡。

颜色从水面方向的青蓝，逐渐过渡为深蓝和藏青。

项目卡片不能继续使用相同的大型矩形。

将项目卡片改成海底世界中的物体，例如：

- 海底档案终端；
- 舷窗；
- 漂浮的探测器屏幕；
- 沉没的数据舱；
- 发光贝壳中的项目记录。

项目文字和按钮必须保持DOM结构。

每张项目卡片仍显示：

- 项目名称；
- 简介；
- 技术栈；
- 项目状态；
- 查看详情。

但边框、背景、标签和按钮要形成完整的海底设备视觉。

项目卡片出现时：

- 从远处缓慢浮近；
- 或随着气泡向上移动；
- 不要只进行透明度淡入。

鼠标移动时：

- 近处气泡产生轻微避让；
- 光束方向产生非常轻微变化；
- 卡片可以有2～4px的像素漂浮；
- 不得造成明显晕动。

## 六、调整不同阶段的整体色调

当前页面最大问题之一是所有阶段都保留相同亮蓝色。

必须建立明显不同的色调：

### 陆地

```
青蓝天空
浅色云朵
绿色草地
暖黄色阳光
红色或橙色点缀
```

### 深海

```
青蓝水面
深蓝中层
藏青海底
青绿色荧光
珊瑚红点缀
```

### 星空

```
接近黑色的深蓝
靛紫
星云紫
冷蓝
暖白星光
```

在以下滚动点截取画面时，必须能够一眼区分：

```
0%：明亮陆地
25%：陆地文章区域
50%：深海项目区域
72%：浪花与翻转过渡
90%：星空区域
```

禁止在星空区域继续以亮青色作为主背景。

## 七、第三阶段：星空世界

星空阶段必须真正变为暗色星空，而不是在亮蓝背景上放一个深色卡片。

必须包含：

1. 深色天空背景；
2. 至少三层不同尺寸和移动速度的星点；
3. 像素星云；
4. 银河带；
5. 少量流星；
6. 星座连线；
7. 远处小行星或漂浮岛；
8. 从上一阶段泡沫转化而来的星尘；
9. 前景漂浮粒子。

“关于我”内容不要只放在一个普通矩形卡片中。

可以设计为：

- 漂浮的星际信号站；
- 太空档案窗口；
- 星球旁的数据面板；
- 星座中央的个人信号；
- 漂浮岛上的终端。

内容仍保持清晰易读。

在星空阶段，文章和项目入口可以形成可点击星座：

- 主星表示文章或项目；
- 悬停显示名称；
- 点击进入详情页；
- 移动端自动降级为普通列表。

陆地阶段：

导航和页面正文不参与场景翻转，始终保持水平和可读。

- 随滚动显示完成进度；
- 桌面端保持右侧；

## 十、性能和实现约束

1. 保留当前可逆滚动架构；
2. 不劫持滚轮；
3. 不锁死用户滚动；
4. 所有重要动画都必须可倒放；
5. Canvas初始化失败时，DOM内容仍然可访问；
6. 支持prefers-reduced-motion；
7. 移动端粒子数量降低；
8. DPR最高限制为2；
9. 页面切后台时暂停环境动画；
10. 页面销毁时清理Ticker、监听器和ScrollTrigger；
11. 不使用全屏视频；
12. 不加入后端；
13. 不在前端写入任何密钥；
14. 不破坏现有文章、项目、搜索和路由功能；
15. 不引入无法静态部署的运行时依赖。

## 十一、Reduced Motion模式

用户开启减少动画时：

- 不执行大幅视差；
- 不执行世界翻转；
- 水位上涨替换为简短像素擦除；
- 深海到星空使用颜色和粒子的短过渡；
- 保留三种不同的静态场景；
- 所有文字、按钮和内容仍然可用。

Reduced Motion不能简单变成纯蓝背景。

------

## 十二、本轮暂不追求的内容

本轮重点是完整场景和视觉层次。

本轮暂时不要求：

- 最终复杂Shader；
- 完整音效系统；
- 最终商业级像素精灵；
- 超高数量粒子；
- 复杂的水体物理模拟。

但是本轮完成后，三个世界必须已经清晰成立，不能继续停留在纯色背景和矩形卡片阶段。

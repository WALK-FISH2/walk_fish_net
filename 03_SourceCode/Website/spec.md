# 产品规格说明书 Spec

版本：1.4.0

状态：当前需求基线  
产品名称：Pixel//Walk（可在站点配置中替换）

## 1. 产品愿景

建设一个具有强烈个人辨识度的纯前端个人网站。访问者可以阅读博文、浏览并体验站点所有者本人编写的程序，并通过陆地、深海、星空三个连续世界了解站点所有者的作品与兴趣。

## 2. 用户

### 访客

招聘者、同行开发者、朋友、文章读者和程序体验者。

### 站点维护者

需要通过 Markdown/MDX 添加文章和程序、更新状态与截图、修改介绍，并使用静态托管上线。

## 3. 信息架构

推荐公开路由：

```text
/                     沉浸式首页
/articles             文章列表
/articles/[slug]      文章详情
/programs             “做点啥呢”程序列表
/programs/[slug]      程序详情与演示
/about                关于我
/404                  404
```

兼容要求：若现有版本使用 `/projects`，迁移后保留 `/projects` → `/programs` 的静态重定向或兼容页。导航显示名称始终为“做点啥呢”。

## 4. 首页叙事

### 阶段 A：明亮陆地

内容：个人名称、一句话介绍、文章入口和精选文章。

视觉：天空、云、山丘、草地、原创像素引导物和前中后景视差。

### 过渡 A：陆地下潜

```text
天空变暗
→ 底部出现水面
→ 水位上涨
→ 卡片经过水线产生短暂折射
→ 水覆盖屏幕
→ 镜头向下
→ 进入海底
```

### 阶段 B：深海程序档案

内容：“做点啥呢”、精选程序、程序状态、技术栈和体验入口。

视觉：深海渐变、光束、气泡、鱼群、海草、珊瑚、水母和海底档案终端。

M6.1 对该阶段的桌面内容层进行独立收尾：进入 Programs 主要阅读窗口后，“沉在海里的程序档案”标题在 section 左上安全区停驻；三张档案卡片整体位于右侧，继续保持当前完整纵向排列和自然滚动，不增加逐张隐藏、接棒或单卡 sticky。最后一张卡片划过时，标题提前从深海亮色变暗并逐渐消失，视觉上隐入即将到来的夜空。标题与卡片、卡片与卡片在正向和反向滚动中都不得重叠。

移动端竖屏保留当前单列顺序：标题在上，三张卡片在下，全部使用普通文档流。中间宽度无法安全容纳双轨时也回退为单列，不得通过压缩造成碰撞。详细规格见 `docs/product/m6-1-underwater-programs-layout-spec.md`。

### 过渡 B：海洋平滑过渡为星空

```text
水流增强
→ 镜头向水面上升
→ 浪花占据屏幕
→ 深海色连续过渡为靛蓝与紫色
→ 泡沫成为星点
→ 气泡成为星尘
→ 水面高光成为银河
→ 星空稳定
```

Canvas 背景世界在全部进度、全部视口和全部动效模式下始终保持水平，不执行旋转或翻转。不得出现船，也不得复刻电影画面。

### 阶段 C：星空

内容：关于我、技能、联系入口、重启旅程和文章/程序入口。

视觉：深色星空、星云、银河、星点、流星、星座和漂浮信号站。

星空内容入口必须恢复并保持项目所有者确认的历史表现：桌面端在信号站左侧或左下安全区常显四行纵向入口，每行由一颗发光菱形像素星和一个可读标题组成；内容为前两篇文章与前两个 Program。四行标题在完整动画和简化动画下都必须默认可见，不得要求 hover 后才出现。hover 与 focus 只允许增强星光、边框或位移，不得控制标题是否存在。移动端改为信号站后的正常文档流列表，保留四项、点击和键盘焦点，不产生横向溢出。

### M6.2：跨世界小章鱼主题角色

首页使用同一个小章鱼主角串联三个世界：

- 陆地使用普通小章鱼替换现有小人，保留横向行程和停止滚动时的轻微上下浮动；
- 陆地进入海洋时，普通小章鱼先向上跃起，再落入现有三层浪花；泡沫遮挡期间切换为潜水小章鱼；
- 潜水小章鱼在深海从屏幕右上逐渐移动到左下，海洋尾段向下潜出屏幕；
- 星空建立后，宇航员小章鱼从屏幕下方升入，在可用视口内执行独立于滚动坐标的 DVD 式漂浮和边界反弹；
- 宇航员位于右上行星和信号站装饰前方、文字卡片和所有交互内容后方，不拦截指针、触控或键盘。

从星空向上返回时，宇航员不能停止反弹，而是在当前反弹轨迹上随滚动逐渐变暗并降低透明度。若尚未完全消失便重新向下，必须在同一轨迹上恢复正常颜色和可见度，不重新入场；若已经完全消失，继续向上时潜水小章鱼从屏幕下方升起，随后再次向下进入星空时，宇航员重新从屏幕下方升入。

M6.2 不修改现有五段全局进度、M4.5 浪花、M5 海空粒子、M5.5 星空视觉、Programs、路由、文章系统和世界固定 `0°`。完整规格见 `docs/product/m6-2-octopus-traveler-spec.md`。

## 5. 文章系统

### 文章列表

- 发布时间倒序；
- 精选文章；
- 标签筛选；
- 标题和摘要搜索；
- 阅读时间；
- 空状态；
- 响应式布局。

### 文章详情

- 标题、摘要、日期、标签；
- 目录；
- Markdown/MDX；
- 代码高亮、图片、表格和引用；
- 上一篇和下一篇；
- SEO 元数据。

正文不强制使用像素字体。

## 6. “做点啥呢”程序系统

### 定义

每个条目必须代表站点所有者本人编写的程序或演示。

程序类别：

```text
web-app
desktop-app
mini-program
tool
visualization
game-prototype
automation
experiment
other
```

程序状态：

```text
completed
in-progress
prototype
archived
```

演示类型：

```text
static-embedded
external-live
video
gif
screenshots
none
```

### 程序列表

显示程序名称、说明、封面、技术栈、类别、状态、演示类型、查看详情、立即体验和可选源码入口。

### 程序详情

包含：

- 程序解决的问题；
- 使用方式；
- 核心功能；
- 本人完成内容；
- 技术方案；
- 关键难点；
- 截图与演示；
- 当前限制；
- 后续计划；
- 源码地址；
- 隐私与数据说明。

### 演示诚实性

若只是前端演示，必须标注：

```text
这是静态演示版。
部分后端、登录、支付、数据库或实时功能未接入。
```

不得模拟成功接口后让访客误以为功能已经完整上线。

## 7. 程序演示安全

站内演示优先采用独立静态路由、沙箱 iframe 或外部新标签页。

iframe 使用最小权限，例如：

```html
sandbox="allow-scripts allow-forms allow-modals"
```

只有确实需要时才添加 `allow-same-origin`、`allow-popups` 或 `allow-downloads`。

## 8. 视觉风格

- 二维高精细像素风；
- 16-bit/32-bit 主机时代质感；
- 可爱但不幼稚；
- 色彩丰富；
- 轮廓清晰；
- 允许渐变、光晕和透明粒子；
- 禁止体素、三维方块和“我的世界”风格。

像素字体用于标题、短标签、按钮和导航；正文使用高可读中文字体。

## 9. 响应式

桌面提供完整视差、平滑海空形态转换、鼠标互动和星座交互；深海 Programs 采用左侧固定并渐隐的标题轨道，以及右侧保持现有完整排列的卡片轨道。

移动端保留三个世界，减少粒子，简化折射，背景同样保持水平；深海 Programs 标题在上、三张程序卡片在下，保持普通文档流，不依赖 hover，不出现横向滚动。M6.2 小章鱼缩小尺寸和活动范围但保留普通 → 潜水 → 宇航员的顺序，优先避开导航、阶段进度、动效开关和主要文字按钮。

## 10. 动效模式与 Reduced Motion

首页提供“完整动画”和“简化动画”两种明确模式：

- 没有 URL 临时覆盖、也没有用户已保存选择时，普通网址默认进入完整动画，不要求访问者添加 `?motion=full`；
- 固定 UI 中提供始终可见、可触控、可键盘聚焦的动效模式开关，用户可以随时切换完整动画和简化动画；
- 用户主动选择写入浏览器本地偏好，刷新、站内跳转和再次访问时继续沿用；本地存储不可用时退化为仅在当前页面会话有效，不得报错；
- `?motion=full` 和 `?motion=reduce` 只作为当前页面的临时覆盖及验收入口，不写入或污染用户已保存偏好；
- 解析优先级固定为：有效 URL 参数 > 用户已保存选择 > 完整动画产品默认值；
- 当系统声明 `prefers-reduced-motion: reduce` 时，站点必须识别该信号，在开关中明确提示“系统建议简化动画”，并允许一次操作切换；根据项目所有者确认，该系统信号不再自动覆盖“首次默认完整动画”的产品策略；
- 已保存为简化动画的用户不得在场景初始化前短暂启动完整动画，避免闪烁、快速视差或动态流星抢先运行；
- 切换模式不改变当前路由、内容顺序、滚动语义或世界方向，不得重建重复的时间线、RAF、timer 或事件监听器；
- 完整与简化模式之间的切换必须保留同一个 `globalProgress`，允许误差不超过 `±0.01`；切换完成后滚轮、触控、触控板、PageDown/PageUp 和阶段节点必须继续有效；
- 任意模式下，页面物理底部必须对应 `globalProgress >= 0.999`、`phase=space` 和可见的星空内容。不得出现滚动条已经到底、全局进度仍停在深海或海空过渡、继续向下输入无效的状态；
- 模式切换期间不得禁用浏览器默认滚动。连续或快速切换时只允许最后一次切换事务恢复位置，旧 RAF、延迟 refresh 或旧坐标不得覆盖新状态。

简化动画开启后：

- 背景保持水平，并取消快速视差；
- 水位上涨改为像素遮罩或短淡入；
- 三个世界依然有不同的静态场景；
- 动态流星及其他高速、非滚动时间动效停止并完成清理；
- M6.2 保留当前章节对应的小章鱼形态与必要的形态切换，关闭持续上下浮动和宇航员 DVD 式边界反弹；
- 内容顺序和功能不变。

## 11. 静态部署

生产构建必须输出静态目录，生成所有文章和程序详情，不需要常驻 Node，支持 GitHub Pages 或 Cloudflare Pages，刷新详情页不报 404。

## 12. SEO

每页提供唯一 title、description、canonical、Open Graph、正确语义结构、sitemap 和 robots.txt。RSS 推荐实现。

## 13. 非目标

第一阶段不实现登录注册、自建评论后端、数据库后台、在线支付、多人实时协作、复杂服务端搜索或必须 SSR 的功能。

## 14. 成功标准

1. 三个世界不看文字也能分辨；
2. 陆地到深海连续；
3. 深海到星空自然；
4. 向上滚动可倒放；
5. 文章可通过 Markdown/MDX 添加；
6. 程序示例均为本人编写；
7. 程序演示有明确状态和限制；
8. 详情页可直接访问和刷新；
9. 静态构建成功；
10. 移动端和简化动画可用，普通网址默认完整动画，用户选择可保存且 URL 临时覆盖不污染偏好；
11. Canvas 失败时内容可访问；
12. 没有密钥和侵权素材；
13. 构建、测试和验收有记录。

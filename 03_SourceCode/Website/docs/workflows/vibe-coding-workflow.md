# Vibe Coding 工作流

本项目中的 Vibe Coding 不是“把一句需求丢给 AI，然后接受所有输出”，而是受文档、任务、测试和架构决策约束的协作流程。

## 1. Define

明确要解决什么、不解决什么、用户可见结果、技术边界和验收标准。更新 `spec.md`、`tasks.md`，必要时更新宪章。

## 2. Inspect

AI 先检查仓库、现有实现、package.json、路由、测试、文档、Git 状态和用户未提交改动。禁止直接生成新项目覆盖现有工程。

## 3. Plan

计划必须包含涉及文件、实施顺序、风险、测试、文档更新和非目标。计划应小到一轮内可验证。

## 4. Implement

坚持最小改动、保留可用功能、不顺手大重构、不引入无关依赖、不伪造测试结果、每个阶段可运行。

## 5. Verify

至少覆盖类型、Lint、测试、构建、浏览器、视觉、路由和文档。

## 6. Record

更新 `tasks.md`、`CHANGELOG.md`、ADR、README 和对应规范。聊天不是长期事实来源。

## 7. Review

检查需求、旧功能、秘密、静态部署、无障碍、移动端和技术债。

## 8. Ship

只有验收满足、发布检查完成、构建产物确认和文档同步后才发布。

## 9. 失败处理

结果不理想时，不立刻推翻全部代码。明确差距、截图标注、写成小任务、限定修改范围，再让 AI 修复。

不要只说“继续美化”，应写成：

```text
当前50%滚动位置仍是亮青色背景。
本轮只重做深海背景和程序终端，不修改文章系统与路由。
验收时50%截图必须包含深蓝渐变、光束、鱼群、气泡、海草和海床。
```
